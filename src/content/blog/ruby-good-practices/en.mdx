This article walks through the practices that changed how I write Ruby the most. Some are pure Ruby, some are Rails — but you'll be able to apply all of them starting from your next PR.

## The Domain We'll Use

So that every example makes sense in context, we'll use the same domain throughout: an order system with three models (`Product`, `Order` and `LineItem`).

```ruby
class Product < ApplicationRecord
  # columns: id, name, price, discount_rate, stock
  has_many :line_items
  validates :name, presence: true
  validates :price, numericality: { greater_than: 0 }
end

class Order < ApplicationRecord
  # columns: id, user_id, status, total, created_at
  belongs_to :user
  has_many :line_items, dependent: :destroy

  scope :pending,   -> { where(status: 'pending') }
  scope :completed, -> { where(status: 'completed') }
end

class LineItem < ApplicationRecord
  # columns: id, order_id, product_id, quantity, unit_price
  belongs_to :order
  belongs_to :product
end
```

Reference tables:
- `products`: `id, name, price, discount_rate, stock`
- `orders`: `id, user_id, status, total, created_at`
- `line_items`: `id, order_id, product_id, quantity, unit_price`

## 1. **Use Descriptive Names**

Variable and method names should reflect exactly what they do. A cryptic name saves two seconds for the person writing it and costs ten minutes for every reader.

```ruby
# Bad
def total(u, d)
  u.orders.sum(&:total) * (1 - d)
end
```

```ruby
# Good
def total_with_discount(user, discount_rate)
  user.orders.sum(&:total) * (1 - discount_rate)
end
```

<Callout type="tip">
  **Code is read far more often than it's written.** A descriptive name replaces the comment — the code documents itself.
</Callout>

## 2. **Use Guard Clauses**

Guard clauses are early returns that eliminate nesting. Instead of wrapping all your logic in an `if`, you bail out early when conditions aren't met.

```ruby
# Bad
def create_order(user, order_params)
  if user.present?
    if order_params.present?
      Order.create(order_params)
    else
      raise 'Order params are required'
    end
  else
    raise 'User is required'
  end
end

# Good
def create_order(user, order_params)
  raise 'User is required' unless user.present?
  raise 'Order params are required' unless order_params.present?

  Order.create(order_params)
end
```

<Callout type="info">
  **.present? is Rails, not pure Ruby.** `.present?` (and `.blank?`) come from ActiveSupport. In pure Ruby use `nil?` or `!value.nil?`. Knowing the difference avoids confusion when switching between plain Ruby and Rails projects. For the `&.` operator that prevents `NoMethodError` directly, see Section 5.
</Callout>

## 3. **Avoid Long Methods**

A long method is hard to read, hard to test, and hard to change without breaking something. The single responsibility principle applies to methods too: each one does exactly one thing.

```ruby
# Bad
def process_order(order)
  order.total = order.line_items.sum { |li| li.unit_price * li.quantity }

  order.line_items.each do |li|
    product = li.product
    product.stock -= li.quantity
    product.save!
  end

  order.status = 'confirmed'
  order.save!
  OrderMailer.confirmation(order).deliver_later
end

# Good
def process_order(order)
  order.total = calculate_total(order)
  decrement_stock(order)
  confirm_and_notify(order)
end

private

def calculate_total(order)
  order.line_items.sum { |li| li.unit_price * li.quantity }
end

def decrement_stock(order)
  order.line_items.each do |li|
    li.product.update!(stock: li.product.stock - li.quantity)
  end
end

def confirm_and_notify(order)
  order.update!(status: 'confirmed')
  OrderMailer.confirmation(order).deliver_later
end
```

## 4. **Use Scopes for Common Queries**

Up to here we've been talking about pure Ruby. Now we move into Rails.

Scopes return an ActiveRecord relation, which means you can chain them with other query methods. Use them for common queries and to keep your code DRY.

```ruby
class Order < ApplicationRecord
  scope :recent,    -> { where('created_at > ?', 1.month.ago) }
  scope :confirmed, -> { where(status: 'confirmed') }
  scope :pending,   -> { where(status: 'pending') }
end
```

Don't forget to also use Query Objects for more complex queries — they keep the code cleaner.
Tip: I usually create a new directory at `app/queries` and put one file per query.

```ruby
# app/queries/orders_query.rb
class OrdersQuery
  def self.recent_confirmed
    Order.recent.confirmed
  end

  # More complex queries
  def self.top_buyers_last_month
    query = <<~SQL
      SELECT users.*, SUM(orders.total) AS total_spent
      FROM users
      JOIN orders ON orders.user_id = users.id
      WHERE orders.created_at > NOW() - INTERVAL '1 month'
      GROUP BY users.id
      ORDER BY total_spent DESC
      LIMIT 10
    SQL

    User.find_by_sql(query)
  end
end
```

<Callout type="warning">
  **Bang methods (!) raise, the others don't.** `order.save` returns `false` on a validation failure — silent. `order.save!` raises `ActiveRecord::RecordInvalid` — explicit. When you combine scopes with writes, you need to decide: do you want to know about errors explicitly (bang) or handle them via the return value?
</Callout>

## 5. **Use the Safe Navigation Operator (&.)**

When a method chain touches something that can be `nil`, Ruby blows up with `NoMethodError`. The safe navigation operator (`&.`) cuts the chain and returns `nil` instead of raising an exception.

```ruby
# Bad — if user.address is nil, it explodes
city = user.address.city
# => NoMethodError: undefined method `city' for nil:NilClass
```

```ruby
# Good — the chain stops and returns nil
city = user&.address&.city
```

<Callout type="danger">
  **Don't overuse &.** `&.` is for cases where `nil` is a valid domain value. If you use it to silence `NoMethodError` everywhere, you're hiding real bugs — the day an unexpected `nil` shows up, you'll have no stack trace telling you where it came from.
</Callout>

## 6. **Always Think About Performance**

90% of Rails performance problems are N+1. Let's start there.

<Callout type="warning">
  **N+1 anti-pattern.** One query per element in the loop. With 100 orders and an `each` that touches `order.line_items` or `order.user`, that's 101 queries.
</Callout>

```ruby
# Bad — N+1
orders = Order.all

orders.each do |order|
  order.line_items.each do |li|
    puts li.product.name   # one query per line_item, per order
  end
end
```

```ruby
# Good — eager loading with includes
orders = Order.includes(line_items: :product)

orders.each do |order|
  order.line_items.each do |li|
    puts li.product.name   # no extra queries, already in memory
  end
end
```

<Callout type="tip">
  **If there's a loop, there's an includes.** Mental rule: if your loop accesses an association, use `includes` or you'll pay the toll.
</Callout>

Other performance practices worth keeping on your radar:

- **Cache**: for data that doesn't change often (`Rails.cache.fetch`).
- **Expensive callbacks**: prefer jobs or services for complex tasks — callbacks are hard to debug.
- **SQL aggregations**: `Order.sum(:total)`, not `Order.all.map(&:total).sum`.

## 7. **Testing as a Way of Life**

Tests aren't a luxury: they let you refactor without fear. In Rails the convention is RSpec, but Minitest is perfectly fine — it's what the Rails core itself uses.

```ruby
# spec/models/order_spec.rb
require 'rails_helper'

RSpec.describe Order do
  describe '#total' do
    it 'sums the subtotals of each line_item' do
      order = Order.new
      order.line_items.build(unit_price: 100, quantity: 2)
      order.line_items.build(unit_price: 50,  quantity: 1)

      expected = (100 * 2) + (50 * 1)

      expect(order.line_items.sum { |li| li.unit_price * li.quantity })
        .to eq(expected)
    end
  end
end
```

<Callout type="tip">
  **A good test explains the code.** The `describe`/`it` block should read like a sentence: "Order#total sums the subtotals of each line_item". If it doesn't read that way, the test name is wrong — not the code.
</Callout>

## 8. **Modern Ruby Idioms**

Two short but very Ruby idioms worth internalizing.

### `||=` for memoization

If an operation is expensive (a calculation, a query), memoizing it with `||=` runs it only once per instance.

```ruby
class Order < ApplicationRecord
  def total_with_taxes
    @total_with_taxes ||= calculate_total_with_taxes
  end

  private

  def calculate_total_with_taxes
    line_items.sum { |li| li.unit_price * li.quantity } * 1.21
  end
end
```

`||=` only assigns if the variable is `nil` or `false`. The first call calculates and stores; subsequent calls return the cached value.

### `# frozen_string_literal: true`

A magic comment at the top of the file. It makes all string literals in that file immutable.

```ruby
# frozen_string_literal: true

class Order < ApplicationRecord
  STATUS_CONFIRMED = 'confirmed'
  STATUS_PENDING   = 'pending'
end
```

<Callout type="info">
  **Why frozen_string_literal matters.** Every time you write `'confirmed'`, Ruby allocates a new string object. With the magic comment, those strings are shared and nobody can accidentally mutate them. It's the default in many Ruby 3+ projects, but the comment is still explicit and portable.
</Callout>

## Conclusion

None of these practices are magic — they're habits. The first time you use `&.` or `||=` it'll feel a bit foreign; by the tenth time it'll come naturally.

If you're just getting started with Ruby, pick one or two of these and apply them in your next PR. Don't try to do them all at once: clean code is a muscle, not a checklist.
